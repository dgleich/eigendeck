## April 5 — Multi-Monitor Presenter

### Architecture
- Detect secondary monitor via Tauri `availableMonitors()`
- Open borderless presenter window sized to secondary monitor
- Speaker view on primary: current slide, next preview, notes, timer
- Communication via Tauri `emitTo`/`listen` events

### macOS challenges
- Physical vs logical pixels (divide by scaleFactor for window positioning)
- `setFullscreen` hides dock/menubar globally → don't use fullscreen
- `setSimpleFullscreen` also hides menubar → don't use
- Solution: borderless window + `NSWindow.setLevel_(25)` via cocoa crate
- Covers menu bar on secondary monitor without affecting primary

### Display mirroring
- Detects mirrored displays via `CGDisplayMirrorsDisplay`
- Auto-disables mirroring before presenting (like PowerPoint)
- Re-enables on presentation end
- Uses `ConfigureForSession` (not `ConfigurePermanently` — that prevented sleep)

## April 10-12 — SQLite Storage Design & Benchmarks

### Design (SQLITE_STORAGE.md)
- Temporal data model: valid_from/valid_to on slides, elements, slide_elements
- Junction table: slide_elements maps elements to slides (sync = one element, multiple rows)
- Elements own their position (in data JSON)
- Assets stored as BLOBs with optional file watching
- WAL mode for fast writes, checkpoint on close

### Benchmarks
- tools/bench-storage.mjs: SQLite vs ZIP vs JSON (50MB presentation)
  - SQLite incremental: 0.4ms vs ZIP: 163ms (400x faster)
- tools/bench-datamodel.mjs: UI operations (250 slides)
  - Slide switch: 0.005ms, drag: 0.065ms, sync 20 elements: 0.52ms
- tools/bench-junction.mjs: junction model performance
  - Synced edit: 0.020ms (O(1) vs O(N) in old model)
- tools/bench-json-convert.mjs: toJSON/fromJSON round-trip
  - 250 slides: 9.6ms round-trip, dedup works
- tools/bench-wal-vs-delete.mjs: WAL vs DELETE journal mode
  - WAL writes 48x faster, reads same
- tools/bench-perf.mjs: repeatable performance test suite with --save

## Architecture Summary (as of April 14)

```
User interaction
  → Zustand store (in-memory, fast)
    → Subscriber detects changes (diff prev/current)
      → Dirty tracking (elements, slides, structural changes)
        → Incremental SQLite writes (close old version + insert new)
          → WAL mode for fast writes
            → Periodic checkpoint (merge WAL into main file)

eigendeck-cli (Rust binary)
  → eigendeck_lib::storage (same code as GUI)
    → SQLite queries
      → Same .eigendeck file

HTML export
  → src/lib/exportCore.mjs (shared between GUI and CLI)
    → GUI: Tauri fs + MathJax pre-render (offline SVG)
    → CLI: Node fs + @mathjax/src (PT Sans font)
```

## Key Decisions Made

1. **SQLite is the only format** — no JSON directory support in the app
2. **Zustand for interaction, SQLite for persistence** — no in-memory-only state
3. **Incremental writes** — never full resync in normal flow
4. **WAL mode** — 48x faster writes, checkpoint on close
5. **All SQLite code in Rust** — no WASM, no Node SQLite
6. **Temporal versioning on everything** — slides, elements, slide_elements
7. **Junction table for sync** — one element, multiple slide_elements rows
8. **Assets as BLOBs** — single file contains everything
9. **File watching via Tauri plugin-fs** — for unpacked assets
10. **eigendeck-cli shares code with GUI** — pub mod storage, same queries
